## 1. 背景与目标

P3 已完成工程化与全量 trace 建设，v0.4.0 已发布。P4 进入**能力度量与回归防护**阶段，核心目标是建立一套可复现、可扩展的 agent 评估框架，用于：

- 量化 ForgeLoop 在真实代码任务上的表现。
- 防止后续重构/功能迭代引入回归。
- 为长期优化（prompt、模型、工具）提供数据依据。

### 1.1 当前状态

- `ForgeLoopAgent` + `ToolExecutor` 已支持 Bash/Edit/Find/Grep/List/Read 等工具。
- `FauxProvider` 可用于 deterministic 测试，但无法评估真实 LLM 能力。
- `SessionCoordinator` 提供了统一的会话入口，适合作为 Eval 的执行驱动。
- 无评估数据集、无评分逻辑、无报告生成。
- CI 已有 `PerformanceGateTests`，但无功能正确性 benchmark。

### 1.2 P4 目标

- 新增 `ForgeLoopEval` target，提供评估基础设施。
- 定义 `EvalCase`、`EvalAssertion`、`EvalResult` 模型。
- 实现 `EvalRunner`：在隔离 workspace 中运行 agent 完成任务。
- 实现 `EvalScorer`：支持文件、命令、文本等多种断言。
- 实现 `EvalReporter`：生成 JSON / Markdown 报告。
- 提供 3-5 个示例 benchmark cases。
- CLI 增加 `forgeloop eval` 子命令。
- 接入 nightly CI workflow，定期回归。

---

## 2. 设计原则

1. **可复现**：每个 `EvalCase` 必须有固定的初始文件状态、输入 prompt、期望断言。
2. **可隔离**：每个 case 在独立临时 workspace 运行，运行后清理，不污染真实文件系统。
3. **可扩展**：断言系统插件化，新增断言类型只需实现一个 scorer。
4. **双模式支持**：
   - `deterministic` 模式：使用 `FauxProvider` 验证框架本身。
   - `llm` 模式：使用真实 provider 评估 agent 真实能力（默认 nightly 使用）。
5. **成本可控**：真实 provider 评估默认只在 nightly/dispatch 运行，PR CI 不触发。
6. **不修改业务逻辑**：Eval 框架只消费 `AgentEvent` 和文件系统状态，不侵入 Agent/Tool 内部。

## 4. 核心模型设计

### 4.1 EvalCase

```swift
public struct EvalCase: Sendable {
    public let id: String
    public let name: String
    public let description: String
    public let prompt: String
    public let initialFiles: [EvalFile]
    public let assertions: [EvalAssertion]
    public let timeout: Duration
    public let tags: [String]
}

public struct EvalFile: Sendable {
    public let path: String
    public let content: String
}
```

### 4.2 EvalAssertion

```swift
public enum EvalAssertion: Sendable {
    /// 文件内容包含指定字符串
    case fileContains(path: String, substring: String)
    /// 文件内容不包含指定字符串
    case fileNotContains(path: String, substring: String)
    /// 文件精确匹配（或 diff 匹配）
    case fileEquals(path: String, expected: String)
    /// 文件存在
    case fileExists(path: String)
    /// 命令退出码符合预期
    case commandSucceeds(command: [String])
    /// 命令输出包含指定字符串
    case commandOutputContains(command: [String], substring: String)
}
```

### 4.3 EvalResult

```swift
public struct EvalResult: Sendable {
    public let caseID: String
    public let passed: Bool
    public let score: Double           // 0.0 ~ 1.0
    public let duration: Duration
    public let assertionResults: [AssertionResult]
    public let traceContext: TraceContext?  // 可选：关联 P3 trace
}

public struct AssertionResult: Sendable {
    public let assertion: EvalAssertion
    public let passed: Bool
    public let message: String
}
```

## 7. 风险点

1. **LLM 非确定性**：真实 provider 评估结果可能波动。缓解：nightly 使用 FauxProvider，真实 provider 评估仅手动触发。
2. **文件系统隔离**：workspace 路径处理错误可能污染用户目录。缓解：严格使用 `FileManager.default.temporaryDirectory` 子目录，并在测试中用 `PathGuard` 校验。
3. **超时与资源占用**：agent 可能无限循环。缓解：每个 case 设置 timeout，nightly 限制总运行时间。
4. **与现有测试耦合**：eval 框架不应依赖 `ForgeLoopCliTests` 的具体实现。缓解：`ForgeLoopEval` 只依赖 public API。
5. **维护成本**：新增 case 需要维护初始文件和期望输出。缓解：提供清晰的 case 编写模板。

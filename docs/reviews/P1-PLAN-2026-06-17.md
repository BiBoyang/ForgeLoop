## 阶段 1：核心并发原语安全（3 项）

### P1-1 修复 `SSEParser` 线程安全

**文件**：`Sources/ForgeLoopAI/SSEParser.swift`

**目标**：`SSEParser` 当前标 `@unchecked Sendable` 但内部状态无同步。改为值类型或加锁。

**推荐方案（值类型）**：
```swift
public struct SSEParser: Sendable {
    private var lineBuffer = ""
    private var event = "message"
    private var dataLines: [String] = []
    private var id: String?
    
    public init() {}
    
    public mutating func ingest(_ text: String) { ... }
    public mutating func ingest(bytes: Data) { ... }
    public mutating func drain() -> [SSEMessage] { ... }
    public mutating func finish() -> [SSEMessage] { ... }
    
    private mutating func handle(line: String) { ... }
}
```

**关键点**：
- 所有方法改为 `mutating`。
- Provider 中 `let parser = SSEParser()` 改为 `var parser = SSEParser()`。
- 移除 `@unchecked Sendable`。

**测试**：
- 现有 Provider SSE 测试应继续通过。
- 新增并发测试：多个任务同时 `ingest` 同一个 `var parser`？实际上值类型副本独立，所以测试重点应是“parser 可被 Sendable 地传递”。
- 更实用的测试：高频率 ingest + drain 不产生乱序或丢失消息。

### P1-2 收紧 `Agent` 的公开可变配置属性

**文件**：`Sources/ForgeLoopAgent/Agent.swift`

**目标**：`apiKeyResolver` / `toolExecutor` / `backgroundTaskManager` / `cwd` / `toolExecutionMode` 是公开 `var` 且无同步。

**推荐方案**：
1. 把运行时不应变更的属性改为 `let`：
   - `toolExecutor`（在 init 后很少变更）
   - `backgroundTaskManager`（通常由外部设置一次）
2. 若必须可变，集中到 `LockedConfig` 或用 actor：
   - 创建私有 `ConfigState` struct + `NSLock`
   - 提供 `setCwd(_:)`、`setToolExecutionMode(_:)`、`setAPIKeyResolver(_:)` 方法
3. `makeLoopConfig` 读取时获得一致快照。

**最小改动版本**：
```swift
private final class LockedAgentConfig: @unchecked Sendable {
    private let lock = NSLock()
    var apiKeyResolver: (@Sendable (String) async -> String?)?
    var cwd: String
    var toolExecutionMode: ToolExecutionMode
    
    init(apiKeyResolver: ..., cwd: String, toolExecutionMode: ...) { ... }
}
```

然后 `Agent` 持有 `private let config: LockedAgentConfig`，公开方法修改它。

**测试**：
- 并发读写 `cwd` / `toolExecutionMode` 不崩溃。
- `makeLoopConfig` 在多次读取时得到一致值。

## 阶段 2：Provider 取消与架构（3 项）

### P1-5 统一 Provider 取消模型，减少 `Task.detached`

**文件**：
- `Sources/ForgeLoopAI/AnthropicProvider.swift`
- `Sources/ForgeLoopAI/OpenAIResponsesProvider.swift`
- `Sources/ForgeLoopAI/OpenAIChatCompletionsProvider.swift`
- `Sources/ForgeLoopAI/GeminiProvider.swift`

**目标**：所有 Provider 用 `Task.detached` 启动 worker，与调用者任务树脱离。

**推荐方案**：
1. 提取通用 Provider 骨架到 `Sources/ForgeLoopAI/BaseSSEProvider.swift`：
   - 负责 `AsyncThrowingStream` 包装
   - `onTermination` 调用 `CancellationHandle.cancel()`
   - 提供 `endWithDone` / `endWithError` / `endAbortedIfNeeded`
2. 各 Provider 只实现 `parseSSE(_:) -> [AssistantMessageEvent]`。
3. 取消检查：原 `options?.cancellation?.isCancelled` 保留，同时检查 `Task.isCancelled`。

**注意**：改动较大，如果风险高，可先阶段 2 只改最小版本：把 `Task.detached` 改为 `Task { ... }` 并添加 `Task.isCancelled` 检查；后续再提取骨架。

**最小改动版本**：
```swift
public func stream(...) -> AssistantMessageStream {
    let out = AssistantMessageStream()
    let worker = Task { [self] in
        // existing body
    }
    options?.cancellation?.onCancel { _ in worker.cancel() }
    return out
}
```

但这样 worker 仍脱离调用者任务树。更正确的做法是用 `AsyncThrowingStream`：
```swift
public func stream(...) -> AssistantMessageStream {
    AsyncThrowingStream { continuation in
        let task = Task { ... }
        continuation.onTermination = { _ in task.cancel() }
    }
}
```

但这会改变 `AssistantMessageStream` 的 API。建议保持 API，内部包装：
```swift
public func stream(...) -> AssistantMessageStream {
    let out = AssistantMessageStream()
    let task = Task { [self] in
        // existing body, push to out
    }
    options?.cancellation?.onCancel { _ in task.cancel() }
    return out
}
```

这样调用者取消外层 Task 时，worker 也会取消（因为 `Task {}` 继承上下文，调用者 Task 取消会级联）。

**测试**：
- 新增测试：调用者 cancel Task 后，Provider worker 停止网络请求。
- 现有 Provider 测试通过。

### P1-6 修复 `FauxProvider` 取消被吞

**文件**：`Sources/ForgeLoopAI/FauxProvider.swift`

**目标**：`emitAbortIfNeeded()` 不检查 `Task.isCancelled`；`try? await Task.sleep` 吞取消异常。

**测试**：

## 阶段 4：后台任务与稳定性（2 项）

### P1-10 后台任务资源上限

**文件**：
- `Sources/ForgeLoopAgent/BackgroundTaskManager.swift`
- `Sources/ForgeLoopAgent/Tooling/BgTool.swift`

**目标**：无默认超时、并发上限、输出上限、清理策略；actor 内 Task 可能阻塞其他 actor 方法。

**改动**：
1. 默认超时 5 分钟，暴露 `timeoutMs` 参数。
2. 最大并发后台任务数（如 8）。
3. 输出写入临时文件或环形缓冲区，限制单任务输出大小（如 64KB）。
4. 任务完成后定期清理或限制保留数量。
5. 将 `ProcessRunner.run` 从 actor 隔离中移出：使用 `Task.detached` 或独立 `Task`（非 actor-isolated）运行，actor 只负责状态登记、取消句柄管理和查询。

**测试**：
- 新增测试：超时任务被 kill、超并发返回错误、大输出被截断。
- 新增测试：启动后台任务后，立即调用 `cancel()` / `status()` 不阻塞或能快速响应。
